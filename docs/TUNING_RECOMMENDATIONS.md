> Key principle: the LLM scorer is the real relevance filter. A `title_exclude` term cuts a
> posting *before* it ever reaches the LLM, so only exclude terms that essentially never
> appear in a SWE/AI/Data title. When in doubt, leave it out and let the LLM score it.

## Location strategy (real bug — needs your decision + a small code change)
Two issues surfaced, both rooted in naive substring matching of `locations_allow`:

1. **Foreign-remote leaks in.** `remote` in the allow-list matches `"India - Remote"`,
   `"London (Remote)"`, etc., so non-North-America roles pass. (One real example: OpenAI's
   only survivor was "Partner AI Deployment Engineer - AWS" in *India - Remote*.)
2. **US cities under-covered.** Before today the allow-list had no US terms, so SF-based AI
   labs (Cohere, Mistral, Perplexity) yielded **zero survivors**. Today's hub list helps, but
   it's still whack-a-mole (any unlisted US city is missed).

**Recommended fix (cleaner model): switch from allow-list to block-list.**
Set `locations_allow: []` (allow everything) and instead `locations_block` the foreign
countries/regions. This passes *all* US/Canada cities automatically and blocks foreign.

But naive substring blocking is unsafe: `"india"` matches `"indiana"`, `"uk"` matches many
words. So pair it with a tiny code change in `filters.py` to match block terms against
**comma/dash-split location tokens** (whole-token match for single words; substring for
multi-word phrases like `"united kingdom"`). Sketch:

```python
import re
...
loc = (posting.location or "").lower().strip()
if loc:
    if profile.locations_allow and not any(a in loc for a in profile.locations_allow):
        return False
    tokens = {t.strip() for t in re.split(r"[,/()|\-]+", loc) if t.strip()}
    for b in profile.locations_block:
        if (" " in b and b in loc) or (b in tokens):  # phrase substring, else whole token
            return False
```
Then a `locations_block` of **countries/regions only** (NOT city names): `india, united kingdom,
ireland, germany, france, spain, netherlands, poland, portugal, italy, switzerland, sweden,
singapore, australia, new zealand, japan, korea, china, taiwan, hong kong, philippines,
indonesia, brazil, mexico, argentina, israel, emea, apac, latam` plus unambiguous foreign
tech hubs `bengaluru, bangalore, hyderabad, pune` (extend as needed).

> ⚠️ **Avoid city names that collide with North America.** Blocking `london` would wrongly cut
> **London, Ontario** (Canada); `paris` would cut Paris, Texas. Stick to country/region names
> and clearly-foreign hubs. The token-aware matcher above already protects single words from
> substring collisions (`india` won't match `indiana`), but it can't disambiguate same-named
> cities — so keep cities out of the block-list.
